# Voice Chat (Speech In + Spoken Replies)

**Status**: Shipped · **Last updated**: 2026-05-31

## Introduction

Typing a workout log into a phone mid-gym is awkward. Hands are sweaty, the keyboard takes over half the screen, and the user is usually one set away from picking the next weight — wrong moment for fine motor input. The web's surface has the same problem at the kitchen table after a session, just less acute: people will speak a full session summary faster than they'll type it.

**Prog Strength** today is text-only on both clients. The agent is a coach you have to type at, and its replies appear silently — fine when you're staring at the screen, awful when your hands are on a barbell. Two pieces are missing:

- **Speech in.** A mic button that turns the user's voice into text in the chat composer. The user holds, talks, releases, and the transcript lands in the input ready to send. Same flow as the keyboard but spoken.
- **Spoken replies.** A toggle that, when on, plays back the agent's response as audio after it finishes streaming. The user can keep their eyes off the phone and still get the answer.

Both pieces are batch in v1 — record the whole user prompt, transcribe it; let the whole agent reply land, then speak it. Real-time / streaming versions of both exist (OpenAI realtime API, Web Audio + streaming Whisper) but they're a much bigger investment for marginal feel improvement. Once the batch loop is good, streaming becomes a v2 if there's appetite.

After this work ships, a lifter can log "bench five sets of five at 185, last set was a grinder" while still racking the bar, and hear back "Logged. That last set looks like a 1RM around 215, up from your 210 baseline two weeks ago" while putting plates away.

## Proposed Solution

A **hybrid** stack:

- **Native speech-to-text on both clients.** iOS Speech framework via `expo-speech-recognition` on mobile; Web Speech API on web. Both are free, run locally with zero upload, and stream partial transcripts as the user speaks. Mediocre on rare jargon but fine for the lifting + nutrition vocabulary we care about.
- **OpenAI TTS for the agent's voice.** A new endpoint on the agent (`POST /speak`) takes the assistant's final text and returns mp3 bytes generated by OpenAI's `tts-1` model. Native TTS on iOS and the browser exists but sounds robotic and doesn't pass the "would I want to listen to this in a gym for 30 seconds" bar; OpenAI's voices do.

The OpenAI API key stays server-side; the agent is the only thing that ever holds it. Clients post text, get audio bytes, play them. No client-side key management, no key-rotation concerns on the mobile binary.

UX shape on both surfaces:

- A **mic button** next to Send. Push-and-hold to record, release to stop. The transcript fills the existing composer — the user can edit before sending. No auto-send (false positives from background noise are worse than the extra tap).
- A **voice-mode toggle** in the chat header. Default off. When on, every completed assistant turn plays back via the speaker.
- A **persona is hardcoded** in v1 — one voice across all users (likely `alloy` for neutrality or `onyx` for the "strength coach" feel, decision deferred to an open question below). User-selectable voices are a follow-up; not worth a settings UI in v1.

The agent's text-side `/chat` stream is unchanged. Voice is purely additive: a mic button feeds into the existing composer, a speaker plays the agent's existing reply text. The persistence pipeline, the chat-sessions API, the title-generation flow — all untouched.

## Goals and Non-Goals

### Goals

- Mobile users can hold a mic button in the chat composer, speak into the phone, release, and see their transcript land in the input. Native iOS speech recognizer; no network call.
- Web users get the equivalent affordance backed by the browser's SpeechRecognition API. Graceful degrade (button hidden) on browsers that don't support it.
- A new `POST /speak` endpoint on the agent accepts `{text, voice?}`, calls OpenAI TTS, and returns mp3 audio bytes for the client to play. Auth via the existing JWT middleware.
- A voice-mode toggle on both clients. When on, every completed assistant turn auto-plays. When off, behavior is unchanged from today.
- Per-user cost guardrail: a daily character cap on `/speak` per user (proposed 50,000 chars/day ≈ $0.75 in v1). Above the cap, the endpoint returns 429 and the client gracefully degrades to text-only for that day.
- The OpenAI API key is held only by the agent; clients never see it, never send it, can't read it from a debug build.

### Non-Goals

- **Streaming STT / streaming TTS.** Batch only in v1. The user pushes-to-talk for the full prompt; the client waits for the whole agent reply before playing audio. Realtime APIs (OpenAI's realtime endpoint, server-sent audio chunks) are a real engineering investment for what feels-like-the-same-thing UX at this scale.
- **User-selectable voice personas.** Hardcoded voice in v1; persona switching with a settings UI is a follow-up. Adding a `chat_voice` column to the users table is cheap, but the settings surface to set it isn't worth the v1 scope.
- **Whisper for STT.** Native recognizers are good enough for v1. We can revisit if accuracy on lifting/nutrition terminology becomes a real complaint; the agent endpoint for `/transcribe` would be a clean follow-up since the auth shape and OpenAI SDK are already in place from `/speak`.
- **Persisting the audio.** Neither the user's spoken prompts nor the agent's spoken replies get stored. The persisted record stays the *text* of each turn (already saved via the chat-sessions write path). Re-listening to a past turn means hitting `/speak` again — fine at our scale.
- **Voicemail-style turns.** No "send the audio file as the message" — every voice-in turn produces a normal text message and goes through the existing `/chat` flow.
- **Spoken titles.** The agent's `/title` generator stays text-only. Spoken history-list previews would be neat but aren't worth the per-tap cost.
- **Background mic / hotword wake.** The mic only records while the button is held. No "Hey Coach" listening loop.
- **Mid-stream interruption / barge-in.** The user can't talk over the agent's audio reply to interrupt it. They have to wait for the audio to finish or tap a stop control.

## Implementation Details

### Agent changes (`prog-strength-agent`)

**New dependency**: `openai` Python SDK. Pinned, installed via `uv add`. The agent already speaks to Anthropic for `/chat` and `/title` — this adds a second LLM-vendor SDK for the audio side.

**New endpoint**: `POST /speak`. Body: `{ "text": "<string>", "voice"?: "alloy" }`. Auth: `authenticate(request, jwt_signing_key)` — same middleware `/chat` uses. Behavior:

- Validate `text` is 1..4000 characters. Reject longer text with 400 (clients should chunk if they want > 4000 chars, but Anthropic replies rarely cross 4000 in this product).
- Validate `voice` against the closed enum of OpenAI's supported voices; default to the hardcoded v1 pick when omitted.
- Enforce a per-user daily character cap (default 50,000 chars/day). Hold an in-memory counter keyed by user_id; reset on the day rollover (UTC). Above the cap, 429.
- Call `openai.audio.speech.create(model="tts-1", voice=voice, input=text, response_format="mp3")` and return the audio bytes verbatim with `content-type: audio/mpeg`.

A new `speak.py` module hosts the OpenAI client + the per-user counter, mirroring how `title.py` and `model_router.py` wrap their respective Anthropic calls. Tests are an `httpx.AsyncClient` smoke check against a stubbed OpenAI client.

**Config additions**:

- `OPENAI_API_KEY` (required). Surfaces a `ConfigError` at startup if unset.
- `OPENAI_TTS_MODEL` (default `"tts-1"`). The `"tts-1-hd"` variant exists at higher cost; default to the cheaper one and let the env override.
- `TTS_VOICE_DEFAULT` (default tentatively `"alloy"`). The hardcoded v1 voice; trivial to change before deploy.
- `TTS_DAILY_CHAR_CAP_PER_USER` (default `50000`).

### Mobile changes (`prog-strength-mobile`)

**New dependencies**: `expo-speech-recognition` (native STT) + `expo-av` (audio playback). Both pull native code; **an EAS dev-client rebuild is required** before any of this works in the simulator or on device. JS-only changes won't surface the features.

**`app.json` additions**:

- `ios.infoPlist.NSMicrophoneUsageDescription` — "Prog Strength uses the microphone to log workouts by voice."
- `ios.infoPlist.NSSpeechRecognitionUsageDescription` — "Prog Strength uses speech recognition to turn spoken workouts into log entries."
- Plugin entry for `expo-speech-recognition` (the lib needs a config plugin to wire up the native permission strings).

**Chat composer (`app/(tabs)/chat/index.tsx`)**:

- Mic button to the left of Send. Push-and-hold gesture (long-press handler) starts recognition; release stops it.
- Partial transcripts stream into the existing `input` state as they arrive — the input fills in real-time, same way the keyboard's dictation works.
- Permission flow on first tap: `requestPermissionsAsync()`, persist nothing locally (system handles it).
- Disabled while `streaming` is true (no recording mid-agent-reply).

**Voice-mode toggle**:

- A speaker icon in the Stack header next to "+ New" / "History". Tapping toggles a `voiceMode` boolean held in component state for v1 (defaults to off; survives within a tab session, resets on app restart — fine for now).
- After each completed stream, if `voiceMode` is true: POST the assistant text to `/speak`, get an mp3 blob, play with `expo-av`'s `Audio.Sound.createAsync({ uri: dataUri })`.
- Visual indicator while audio is playing: small animated waveform inside the message bubble's metadata row.

### Web changes (`prog-strength-web`)

**No new deps**. Web Speech API and HTMLAudioElement are browser-native.

**Chat composer (`app/(app)/chat/page.tsx`)**:

- Mic button matching the mobile placement. Push-and-hold via `onMouseDown`/`onMouseUp` (and the `onTouchStart`/`onTouchEnd` for touch screens).
- Use `window.SpeechRecognition || window.webkitSpeechRecognition` — Firefox has limited support; feature-detect at module load and hide the button entirely on unsupported browsers.
- Same partial-transcript-streams-into-input shape as mobile.

**Voice-mode toggle**:

- Same speaker icon in the chat page header.
- After each completed stream, if voice mode is on: fetch `/speak`, `URL.createObjectURL(blob)`, instantiate an `Audio()`, play.

### Cost shape

At v1 caps (50k chars/user/day, ~$0.75/user/day worst case), a single user can't move the needle. The bigger concern is a misbehaving client looping `/speak` calls — the per-user counter caps the blast radius at the configured number regardless of bug.

`tts-1` is the cheapest TTS model (~$15/1M chars). `tts-1-hd` is roughly double; voice quality is noticeably better. Default to `tts-1`; the env override is the escape hatch when someone listens to the difference and decides it's worth it.

Whisper would cost ~$0.006/min of audio IF we ever flip on `/transcribe`. At ~3 voice prompts/day of 10s each ≈ $0.01/day. Negligible.

### Backfill or Migration

None. New endpoint on the agent, new UI on both clients, no schema changes, no user-facing migration. The hardcoded voice is in agent config — change once, deploy, everyone hears it.

## Open Questions

1. **Which hardcoded voice?** OpenAI's six TTS voices for `tts-1`: `alloy` (neutral, slightly female), `echo` (neutral male), `fable` (warm British), `onyx` (deep male, "coach" energy), `nova` (bright female), `shimmer` (soft female). Tentative lean: **`onyx`** — fits the "strength coach" persona and reads as authoritative when delivering training advice. Easy to flip via the env var if it grates.
2. **Push-to-talk vs tap-to-toggle?** Push-to-talk feels right for short prompts (<30s) and avoids the "did I forget to stop recording?" trap. Tap-to-toggle is gentler for longer monologues ("here's everything I did this morning..."). Tentative lean: **push-to-talk for v1**. Revisit if users want to log multi-exercise sessions in one breath.
3. **Should voice mode persist across app restarts?** A user-preference column on `users` would make it sticky, but it's also one more thing to migrate + plumb through both clients. Tentative lean: **session-only for v1**; promote to a persisted preference once the SOW lands and we see whether users actually use it.
4. **Web browser compat strategy?** SpeechRecognition is Chromium-native + works on Safari; Firefox has limited support. Tentative lean: **feature-detect + hide the mic button on unsupported browsers** rather than ship a polyfill. Firefox users keep typing; nothing breaks.
5. **Auto-stop on silence, or strict push-to-release?** Auto-stop is gentler but adds a tunable silence threshold (250ms? 1s?) that we'd have to calibrate. Tentative lean: **strict push-to-release for v1**. The button visibly being held is its own clear contract; silence detection is a v2.
6. **Should the chat list show "🔊" or a similar marker on assistant turns the user listened to?** Could be useful for "which turns have I actually consumed in voice mode" but is mostly trivia. Tentative lean: **no marker in v1**.
